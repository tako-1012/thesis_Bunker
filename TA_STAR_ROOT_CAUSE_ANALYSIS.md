# TA*失敗の真の原因：ノード展開上限の到達

**作成日**: 2026年1月29日  
**分析対象**: dataset3_tastar_final_results.json（10失敗シナリオ）

---

## 🔴 根本原因：ノード展開数の上限（500,000）に達した

### 失敗の定義

失敗した10シナリオ全てで以下の特性を示しています：

```
nodes_explored: 500,000（全て固定）
success: False
path_length: 0.0
```

**これは、TA*の探索がノード上限に到達し、経路探索を完了できなかった**ことを意味します。

## 🔍 TA*の実装コード分析

### plan_path メソッドの関連部分

```python
def plan_path(self, start, goal, voxel_grid=None, max_iters=500000, timeout=None):
    # ... 初期化 ...
    
    while open_heap and nodes < max_iters:  # ← この上限チェック
        if timeout and time.time() - t0 > timeout:
            break
        _, _, current = heapq.heappop(open_heap)
        nodes += 1
        
        if current == g:
            # ゴール到達 → 経路を返す
            return {...success: True, path: [...], nodes_explored: nodes}
        
        # 邪魔なノード処理
        # ...
    
    # ← ここに到達 = max_iters に達した or タイムアウト
    return {...success: False, path: [], nodes_explored: nodes}  # 失敗を返す
```

### 失敗条件

**条件1**: `nodes >= 500,000`に達した → ループを抜ける

**条件2**: 目標に到達できなかった → `success: False` で返す

---

## 💡 失敗の理由

### 1. TA*の計算複雑性が高い

TA*は単純なA*に以下を追加：

- **地形複雑度マップ**: 各ノードで地形コスト計算
- **動的コスト関数**: ノード毎に `adjusted_terrain_cost` を計算
- **プルーニング機構**:
  ```python
  dist_to_goal_raw > straight_distance * prune_distance_factor  # 距離制限
  tentative_g > straight_distance * cost_limit_factor           # コスト制限
  ```

これらのオーバーヘッドにより、**ノード展開数が増加**

### 2. 400×400マップの規模

- **ボクセル数**: 400 × 400 × h = 160,000+ ボクセル
- **隣接ノード**: 各ノードから最大27個の隣接ノード（3D）
- **探索空間**: 指数的に増加

→ 単純なA*では数千ノードで解を見つけるが、**TA*では500kノードに達しても解を見つけられない**

### 3. 地形適応のペナルティ

```python
adjusted_terrain_cost = 1.0 + terrain_weight * (terrain_cost - 1.0)
```

地形が複雑だと、このコスト係数が大きくなり、**ヒューリスティック関数の効率が低下**
→ より多くのノード展開が必要

## 📈 成功シナリオとの比較

### Category 1（150×150）・Category 2（250×250）での成功

- ボクセル数が少ない
- 探索空間が小さい
- 500,000ノード上限に到達せずに解を発見

### Category 3（400×400）での失敗（10/30）

- ボクセル数が多い
- 探索空間が広大
- **500,000ノード上限に到達 → 解を見つける前にループ終了**

## 🔧 改善案

### 短期対策

1. **max_itersを増やす**
   - 現在: 500,000
   - 提案: 1,000,000 以上
   - リスク: 計算時間が倍増、メモリ圧迫

2. **プルーニングを厳しくする**
   ```python
   prune_distance_factor = 1.5  # より早期に候補を絞る
   cost_limit_factor = 2.0       # コストオーバーの候補を削除
   ```
   - リスク: 最適解を見落とす可能性

3. **地形重みを下げる**
   ```python
   terrain_weight = 0.3  # 現在の設定を確認して調整
   ```
   - 効果: ノード展開が減るが、地形適応のメリットが減少

### 長期改善

1. **階層的A*の導入**
   - 大規模マップを分割して探索
   - 各層で異なるグリッド解像度を使用

2. **ビーム探索の活用**
   - 各ステップで上位k個のノードのみを保持
   - ノード数の削減

3. **RRT*などのサンプリング法への切り替え**
   - 小規模マップ: A*/TA*
   - 大規模マップ: RRT*

## ✅ 結論

### 学術的評価

**TA*のスケーラビリティ課題は設計的な限界である**

- 地形適応ロジックにより計算複雑性が増加
- 大規模環境では探索空間の爆発的増加に対応不可
- 500,000ノード上限は実装上の安全装置

### 卒論への記載

```
TA*は小中規模環境（150×150, 250×250）で100%成功率を達成したが、
大規模環境（400×400）では33%の失敗率を示した。失敗の原因は、
地形適応機構による計算複雑性の増加に伴い、ノード展開数が
規定上限（500,000）に到達し、経路探索が完了できなかったためである。

改善には、プルーニング戦略の最適化や階層的探索アルゴリズムの
導入が必要と考えられる。
```
